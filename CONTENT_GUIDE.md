# コンテンツ執筆指針

統計検定準1級 学習サイトの記事を書く・書き直す際のルールと注意点。

---

## 0. 対象読者の想定

**中学〜高校レベルの数学素養を持つ、統計初学者。**

- 微積分の記号は使ってよいが、初出時に「〜を計算する操作」と説明する
- 行列・線形代数は前提知識とせず、必要なら「行列（縦横に数字を並べた表のこと）」レベルから
- 「当然わかるよね」という飛躍をしない
- ページ内で一度も使っていない言葉・記号を突然使わない
- 高校の教科書を読む感覚でスラスラ読める文章を目指す

---

## 1. 基本原則：「具体 → 抽象」の順序

**悪い例**
> 正規分布の確率密度関数は $f(x) = \frac{1}{\sqrt{2\pi\sigma^2}} \exp(...)$ で表される。

**良い例**
> 日本人男性の身長を1万人分測ると、170cmあたりが最も多く、遠ざかるほど減る「鐘の形」になります。
> この形を **正規分布** と呼び、数式で表すと $f(x) = ...$ になります。

**ルール：** 毎セクション、必ず「日常の具体例・場面」から始め、そこから数式・定義へ進む。

---

## 2. セクション構造のテンプレート

各セクションは以下の順番で構成する：

```
① なぜこの話をするのか（1〜2文：前のセクションとの接続、または問いかけ）
② 具体例・日常の場面（「〜を想像してください」「〜という場面で…」）
③ 直感的な説明（数式なしで概念を理解させる）
④ 正式な定義・数式（直感の言語化として提示）
⑤ 試験ポイント（🎯）または補足（📖）
```

**セクション間の橋渡し文は必須。** 前のセクションの結論と次のセクションの問いを繋ぐ。

**良い橋渡し例：**
> 正規分布の形はわかりました。では、「そもそもなぜ手元の標本が母集団に近いと言えるのか？」——この疑問に数学的な保証を与えるのが次の2つの定理です。

## 3. 用語・記号の扱い

### 専門用語
- **初出時に必ず定義する**。括弧内に平易な言葉で補足。
- 英語名がある場合は日本語の後に併記（例：**不偏性（Unbiasedness）**）。
- 読み仮名が必要な語には添える（例：**歪度（わいど）**、**尖度（せんど）**）。

### 数学記号
- 初出時に必ず意味を説明する（例：「$\theta$（シータ）は真の値を表します」）。
- $\hat{\theta}$ のような派生記号も「推定値を表す記号」と明示する。
- 収束記号（$\xrightarrow{p}$、$\xrightarrow{d}$）は使った直後に日本語で意味を説明する。
- **特に注意：** 確率収束・分布収束などの「収束の種類」はページ内で初出時に必ず定義する。前提知識として省略しない。

---

## 4. 数値・方向性のチェック

**数値の「意味」と「説明文」が一致しているか必ず確認する。**

- ❌ 間違い：「99.7%——滅多にないレアケース」　→　✅ 正しい：「99.7%——ほぼ全員がここに収まる（外は0.3%のレアケース）」
- ❌ 間違い：「68.3%——珍しい」　→　✅ 正しい：「68.3%——だいたいのひとはここ」

ルール：**割合が大きい＝多い／普通、割合が小さい＝珍しい**。

---

## 5. 絵文字マーカーの役割

- **💡**：概念の導入・直感的な説明・具体例（数式の羅列には使わない）
- **📖**：補足・発展的内容・証明のヒント（メイン説明の代わりにしない）
- **🎯**：試験頻出・必ず覚える数値や定義（単なる重要事項には使わない）

---

## 6. 深さの基準

準1級の内容は難度が高いため、**各セクションは最低でも5〜8文の本文**を書く。

- ❌ 定義と公式だけを箇条書き　→　✅ 日常の例→直感説明→定義→数式→試験ポイントの流れ
- ❌ 1〜2行で終わるセクション　→　✅ 各セクション 100〜200 文字以上の本文
- ❌ 「〜です。」で終わる　→　✅ 「なぜそうなるか」「どう使うか」まで書く

**良い例（CLTの説明）：**
> サイコロを1回振っただけでは、結果（1〜6の整数）は均等に出るだけで「鐘の形」にはなりません。
> しかし、30個のサイコロを同時に振って合計を記録することを1000回繰り返すと、その合計の分布は
> 正規分布に近い形になってきます。この「平均の分布が正規分布に収束する」性質が**中心極限定理**です。
> 元の分布が何であれ——一様分布でも指数分布でも——サンプルサイズ $n$ が大きくなると
> 標本平均の分布は $N(\mu, \sigma^2/n)$ に近づきます。

## 7. 各モジュールの冒頭 💡 ブロック

各モジュールは必ず以下の構成で始める：

```
💡 **このモジュールで学ぶこと**

「〇〇という場面を想像してください」——具体的な問いかけ（1〜2文）
→ この問いに答えるのが [モジュールのテーマ] です。
→ このモジュールを学ぶと [何ができるようになるか] がわかります。
```

---

## 8. 発展的な内容（MGF・定理・証明など）の扱い

発展的なトピックは以下の順で書く：

```
① 「なぜこの道具が必要か」を先に述べる（動機付け）
② 📖 マーカーで「発展」と明示する
③ 数式
④ 具体的な使い道（「〜を証明するときに使う」）
```

**悪い例：** モーメント母関数の公式を突然出して「t=0で微分すると期待値が求まります」で終わる。

**良い例：**
> 「2つの独立な確率変数の和の分布を求めるには、直接計算すると畳み込み積分が必要で非常に複雑です。
> そこで使う道具がモーメント母関数（MGF）です。正規分布では：$M_X(t) = ...$」

---

## 9. 話題の配置：文脈と近い場所に置く

補題や定理は、**内容的に最も近いトピックの直後**に配置する。末尾に「おまけ」として追加しない。

- スルスキーの定理 → 大数の法則・CLTの直後（収束の話の流れ）
- クラメール-ラオの下界 → 有効性の説明の直後

---

## 10. 数式の書き方ルール（技術的制約）

`modules.ts` は TypeScript の文字列なので、**バックスラッシュを必ず二重にする**。

```
✅ 正しい：  \\sigma、\\frac{a}{b}、\\bar{X}、\\exp
❌ 間違い：  \sigma、\frac{a}{b}、\bar{X}、\exp
```

### インライン数式 vs ブロック数式
- `$ ... $` — インライン（文章の中）
- `$$ ... $$` — ブロック（数式を独立した行で表示）

### 禁止事項
- σ、μ、χ、±、ρ などの記号を `$ $` の外でそのまま使わない
- バックスラッシュ1本の `\sigma` などは**バリデーターエラー**になる

**記事を書き終えたら必ず `npm run validate` を実行**してエラー0件を確認する。

## 11. インタラクティブグラフの使い方

`[[interactive:TYPE]]` タグをコンテンツ内に埋め込むと、その位置にグラフが表示される。

### 利用可能なタイプ一覧

| タグ | 表示内容 | 主な用途 |
|-----|---------|---------|
| `[[interactive:normal]]` | CLT：n を増やすと正規分布に収束する様子 | 中心極限定理 |
| `[[interactive:t]]` | t分布と正規分布の比較（自由度スライダー） | t検定・小標本 |
| `[[interactive:chi2]]` | χ²分布（自由度スライダー） | 適合度・独立性検定 |
| `[[interactive:f]]` | F分布（自由度スライダー） | 分散分析 |
| `[[interactive:skewkurt]]` | 歪度・尖度の違いを視覚化（ファセット表示） | 分布の形の説明 |
| `[[interactive:update]]` | ベイズ更新：事前→事後分布の変化 | ベイズ統計 |
| `[[interactive:overfit]]` | 過学習：モデル複雑さと汎化性能 | 機械学習・モデル選択 |
| `[[interactive:multico]]` | 多重共線性：相関と推定の不安定さ | 重回帰分析 |
| `[[interactive:regression]]` | 回帰直線 | 単回帰・重回帰 |
| `[[interactive:logistic]]` | ロジスティック回帰 | 分類問題 |
| `[[interactive:pca]]` | 主成分分析 | 次元削減 |
| `[[interactive:outlier]]` | 外れ値・距離の可視化 | 外れ値検出 |
| `[[interactive:mcmc]]` | MCMC サンプリング | ベイズ推定 |
| `[[interactive:gibbs]]` | ギブスサンプリング | 多変量ベイズ |

### グラフを使うときのルール

- タグの直前に「何を見るべきか」を必ず書く。スライダーを動かして何が起きるかを事前に予告する。
- グラフの後に「このグラフから分かること」を1〜2文で補足する。
- インタラクティブが不要なトピック（推定・検定の理論など）には無理に使わない。

## 12. クイズの設計指針

各モジュールに **必ず10問** 必要（バリデーターで強制）。

- **基礎確認 3問**：定義・用語の意味を問う。モジュールの最初の節に対応。
- **応用 4問**：計算・手順の適用。「〜のとき〜はどうなるか」形式。
- **発展 3問**：複数の概念の組み合わせ、試験頻出の引っかけ問題。

各問のルール：
- `question`：何について問うかを具体的に書く（「次のうち正しいものはどれか」だけは不十分）
- `options`：4択。正解1つ、ダミー3つはそれぞれ「よくある誤解」に対応させる
- `explanation`：正解の**理由**を書く。「〜だから正解」ではなく「〜という考え方をするので〜になる」まで踏み込む
- 数式は `$ $` で囲む（バックスラッシュ二重ルールはここでも適用）

## 13. 改訂前チェックリスト

記事を書き終えたら以下を確認する：

- [ ] 各セクションは「具体例」から始まっているか
- [ ] セクション間に「なぜ次の話をするのか」の橋渡し文があるか
- [ ] 初出の専門用語・記号に定義が添えられているか
- [ ] 収束の種類（確率収束・分布収束）は初出時に定義されているか
- [ ] 数値と説明文の方向性（多い/少ない）が一致しているか
- [ ] 各セクションが5文以上・100文字以上の本文になっているか
- [ ] 発展的な内容は📖マーカー付きで動機を先に書いているか
- [ ] インタラクティブグラフの前後に「何を見るか」の案内文があるか
- [ ] クイズが10問あり、基礎3・応用4・発展3のバランスになっているか
- [ ] マークダウンテーブル（`|---|`）を content 内で使っていないか
- [ ] `npm run validate` がエラー0件か（ダブルバックスラッシュ・数式など）
